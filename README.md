# Data Reconciler

Инструмент сверки данных между произвольными источниками. Подключил два датасорса, настроил маппинг, получил расхождения — без AI, чисто алгоритмически.

---

## Зачем

BI-аналитик работает с данными из разных систем: 1С, ClickHouse, PostgreSQL, Excel. Данные перетекают из одной системы в другую через ETL-пайплайны, и по дороге неизбежно ломаются: строки теряются, суммы расходятся, даты сдвигаются, дубликаты появляются из-за retry.

Сейчас аналитик ищет эти расхождения руками — пишет SQL-запросы к обоим источникам, сводит результаты в Excel, глазами ищет где не сходится. Это занимает 60-70% рабочего времени. Особенно больно в финансах, где допустимое расхождение — ноль. Один тенге дельты в 910 форме — это уже ошибка, по которой неправильно начисляют ИПН.

Data Reconciler автоматизирует именно эту часть: подключился к двум базам, сказал "таблица А здесь = таблица Б там, сверяй по такому ключу" — получил отчёт с точным списком расхождений. Не "примерно тут что-то не так", а "строка id=13, поле total_amount, в 1С 279980, в ClickHouse 289980, дельта -10000".

---

## Для кого

Основной пользователь — BI-аналитик, не разработчик. Он:

- Знает SQL, но не хочет писать сверочные запросы руками каждый раз
- Работает с 1С, ClickHouse, PostgreSQL, иногда Excel/CSV
- Не имеет доступа к ETL-пайплайнам (это делает "инженер"/разработчик)
- Хочет точно знать: "клик надёжный или нет" — то есть данные в аналитической БД соответствуют источнику истины
- Не готов ставить Python/Docker/CLI-утилиты — ему нужна одна кнопка

## Типичные сценарии

**Склад и поставщики.** Три таблицы: факт движения, факт остатков, факт продаж. Балансовое правило: остаток вчера + приход - расход - продажи = остаток сегодня. Если не сходится — баг в ETL или в формуле аналитика.

**Финансовая отчётность.** 910 форма (доходы ИП/ТОО для начисления ИПН). Данные из 1С должны 1:1 совпадать с тем, что попало в аналитику. Порог допустимости расхождений — ноль.

**Маркетинг.** Лиды из CRM vs. данные в аналитике. Какие каналы просели, где потерялись записи при импорте.

**Закупки.** Сколько заказали у поставщика, сколько произведено, сколько отгружено на склад — сверка между системой поставщика и собственной учётной системой.

### Почему Go

- Компилируется в один бинарник — отдать аналитику один .exe файл, без зависимостей
- Cross-compile из коробки: `GOOS=windows go build` и готово
- `database/sql` — стандартный интерфейс для всех SQL-баз, драйверы подключаются импортом
- Фронт встраивается через `embed` — один бинарник = и бэкенд, и фронт
- Хорошая производительность для построчной сверки больших таблиц

### Почему единый DataSource интерфейс

Ключевое архитектурное решение — все источники данных реализуют один и тот же интерфейс:

```go
type DataSource interface {
    Connect(ctx context.Context) error
    Close() error
    Ping(ctx context.Context) error
    GetTables(ctx context.Context) ([]TableInfo, error)
    GetSchema(ctx context.Context, table string) ([]ColumnInfo, error)
    Query(ctx context.Context, q QuerySpec) (*QueryResult, error)
    CountRows(ctx context.Context, table string, where string) (int64, error)
    GetAggregates(ctx context.Context, ...) (*QueryResult, error)
}
```

Reconciler Engine работает только с этим интерфейсом. Ему всё равно, PostgreSQL там или ClickHouse — он получает строки и сравнивает.

### Драйверы

Go `database/sql` даёт единый API для всех SQL-баз. Разница между драйверами минимальна:

| БД | Драйвер | DSN формат | Quote | Дефолт порт |
|----|---------|------------|-------|-------------|
| PostgreSQL | `github.com/lib/pq` | `host=... port=... dbname=...` | `"field"` | 5432 |
| ClickHouse | `clickhouse-go/v2` | `clickhouse://user:pass@host:port/db` | `` `field` `` | 9000 |
| MySQL/MariaDB | `go-sql-driver/mysql` | `user:pass@tcp(host:port)/db` | `` `field` `` | 3306 |
| SQLite | `mattn/go-sqlite3` | путь к файлу | `"field"` | — |
| MSSQL | `microsoft/go-mssqldb` | `sqlserver://user:pass@host:port?database=db` | `[field]` | 1433 |

Вся разница инкапсулирована в структуре `SQLDialect`:
- `BuildDSN` — как собрать строку подключения
- `QuoteIdent` — как экранировать имена таблиц/полей
- `ListTablesSQL` — SQL-запрос для получения списка таблиц

Добавить новую базу = написать ~30 строк нового диалекта. Основной код сверки не трогается.

### Не-SQL источники (будущее)

Тот же интерфейс `DataSource` будет реализован для:
- **1С** через OData REST API (1С умеет отдавать данные по HTTP)
- **Excel/CSV** через Go-библиотеки парсинга (excelize, encoding/csv)
- **Google Sheets** через Google Sheets API
- **REST API** — универсальный коннектор для произвольных HTTP-источников

Пользователю без разницы что под капотом. Он в UI выбирает тип, вводит креды — работает.

## Три режима сверки

### Count
Самый простой. Количество строк в таблице A = количество строк в таблице B. Если нет — ETL потерял или задублировал записи.

### Aggregate
Сравнение агрегатов: SUM(amount) в A = SUM(amount) в B. Быстро для больших таблиц — не нужно тащить все строки. Но не показывает *какие именно* строки расходятся.

### Row-level
Построчное сравнение по ключу. Самый точный, но самый тяжёлый. Для каждой строки в A ищет соответствующую строку в B по ключевым полям, сравнивает все маппированные поля. Находит:
- Потерянные строки (есть в A, нет в B и наоборот)
- Дубликаты (несколько строк с одним ключом)
- Расхождения в значениях (с точной дельтой)

### Balance check (запланирован)
Специальный режим для складских остатков: проверяет что `остаток_вчера + приход - расход = остаток_сегодня`. Не сравнение двух источников, а проверка внутренней консистентности.

## Доставка пользователю

### Этап 1: Go бинарник + браузер (сейчас)
Аналитик скачивает один файл, запускает, открывает `localhost:8080` в браузере. Максимально просто, ноль зависимостей.

### Этап 2: Electron обёртка (если зайдёт)
"Настоящее" десктопное приложение с иконкой в трее. Electron запускает Go как child process. В debug-режиме — просто `loadURL('http://localhost:8080')`, Go запущен отдельно.

### Этап 3: Production Electron (если продукт)
Embedded Go бинарник внутри Electron, автоапдейт через GitHub Releases, installers под Windows/macOS/Linux.

### Автообновление
При запуске бинарник проверяет GitHub Releases API — если есть новая версия, предлагает обновиться. Два бинарника (бэк и фронт) обновляются независимо: фронт — замена статики, бэк — замена .exe.

## UI

### Визуальный query builder
В UI должна быть возможность не только настроить сверку, но и просто посмотреть данные: выбрать таблицу, выбрать поля, добавить JOIN перетаскиванием связи между таблицами, отфильтровать. Без написания SQL. Как в Metabase — аналитик строит запрос мышкой.

Набор операций:
- Выбор таблицы
- Выбор полей (SELECT)
- Фильтрация (WHERE) — через dropdown: поле, оператор, значение
- Агрегации (SUM, COUNT, AVG, MIN, MAX)
- GROUP BY
- Визуальный JOIN — два столбца таблиц, перетаскивание связей между полями
- ORDER BY
- LIMIT

### Маппинг полей
Два столбца: поля Source A слева, поля Source B справа. Пользователь соединяет поля которые соответствуют друг другу. Если имена совпадают — автоматически. Если нет (quantity ↔ qty, amount ↔ sum) — вручную.

### Результаты
Главный экран. Зелёное/красное. Drill-down до конкретной строки. Экспорт в Excel. Числа с разделителями тысяч. Дельта со знаком. Всё как аналитик привык видеть.

## Тестовые данные

Docker Compose поднимает PostgreSQL (имитация 1С, источник истины) и ClickHouse (аналитика после ETL). Данные — розничная торговля электроникой в Казахстане: Samsung, Xiaomi, 20 SKU, 5 поставщиков, ~90 продаж за январь 2025.

В ClickHouse намеренно заложено **8 типичных ETL-багов**:

| # | Тип | Что случилось |
|---|-----|---------------|
| 1 | Потеря строки | Приход Samsung TV 15шт потерян при ETL |
| 2 | Потеря строки | Приход кондиционеров 35шт потерян |
| 3 | Дубликат | Продажа ЧК-0010 записана дважды (retry) |
| 4 | Ошибка суммы | total_amount 279980 → 289980 |
| 5 | Сдвиг даты | Дата движения 2025-01-08 → 2025-01-06 |
| 6 | Кеш прайса | unit_price 89990 → 89900 (устаревший) |
| 7 | Ошибка расчёта | Остаток quantity 75 → 73 |
| 8 | Пустое поле | customer_type '' вместо 'retail' |

Именно такие баги аналитик ловит руками в реальной работе. Инструмент должен найти все 8 автоматически.

## Конкуренты и отличия

| Инструмент | Для кого | Проблема |
|-----------|----------|----------|
| dbt tests | Data engineers | Нужен dbt, Python, CLI. Аналитик не потянет. |
| Great Expectations | Data engineers | Overkill для простой сверки двух таблиц. |
| Soda Core | Data engineers | Опять CLI + YAML. |
| Metabase | Аналитики | Визуализация, не сверка. |
| Excel руками | Аналитики | Работает, но 60% времени уходит на это. |

Data Reconciler — инструмент для аналитика, не для инженера. Один бинарник, визуальный UI, никакого CLI/Python/YAML. Подключил → настроил мышкой → получил результат.

## Как запустить

```bash
# 1. Тестовые БД
docker-compose up -d

# 2. Go backend
go mod tidy
make run

# 3. Открыть
open http://localhost:8080
```

PostgreSQL: `localhost:5432`, db=source_of_truth, user=recon, password=recon123
ClickHouse: `localhost:9000` (native) / `localhost:8123` (HTTP), db=analytics, user=recon, password=recon123
